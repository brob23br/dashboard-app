Overview

This demonstration will showcase the process of creating and using an SSH key pair to establish a secure connection between two Linux computers. The key points of this demonstration include:

  • Generating an SSH key pair on the client machine
  • Transferring the public key to the server machine
  • Testing the SSH connection using the new key pair

Benefits of SSH Key-Based Authentication

SSH key-based authentication offers several advantages over traditional password-based authentication:

  • Enhanced Security: SSH keys are significantly more difficult to compromise compared to passwords, providing stronger protection against brute-force attacks.
  • Improved Convenience: Once set up, key-based authentication eliminates the need to enter passwords for each login, streamlining the connection process.
  • Automated Processes: Key-based authentication facilitates automated scripts and processes that require secure, passwordless connections.
  • Centralized Key Management: Public keys can be easily distributed and managed across multiple servers, simplifying access control for system administrators.
  • Two-Factor Authentication: SSH keys can be combined with passphrases, creating a two-factor authentication system for added security.
  • Reduced Risk of Credential Theft: Since no password is transmitted during the authentication process, the risk of credential interception is minimized.

By implementing SSH key-based authentication, users can significantly enhance the security and efficiency of their remote connections, making it a preferred method for many system administrators and developers.

Demo

  1. I will be using Kali Linux to create the ssh keys. Open terminal.
  1. The ssh-keygen command is used to generate SSH key pairs. Here's a brief overview of the command and its common flags:

    • Basic syntax: ssh-keygen -t rsa -b 4096 -C "your_email@example.com"

    • -t: Specifies the type of key to create (e.g., dsa, rsa, ed25519)

    • -b: Sets the key length in bits (e.g., 4096 for RSA)

    • -C: Adds a comment to the key (often an email address)

    • -f: Specifies the filename of the key pair

    • -N: Sets a new passphrase for the private key

    • -p: Changes the passphrase of an existing private key

    Now, let's generate an SSH key pair using the ssh-keygen command:

  1. Input the command ssh-keygen -t rsa -b 4096
  • You will be prompted on where to save the file. Press ENTER to save to the default path.
  1. Next, you will be prompted to enter a super secret passphrase.
  1. Once the passphrase is entered, the key will be created.
  1. Now we need to copy the public key to the host you wish to connect to. This will be done using the ssh-copy-id command.

The ssh-copy-id command is a utility used to copy your public SSH key to a remote server's authorized_keys file. Here's a brief summary of the command:

  • Syntax: ssh-copy-id [-i [identity_file]] [user@]host
  • Purpose: Automates the process of adding your public key to a remote server for passwordless SSH access
  • Benefits: Simplifies key deployment and reduces the risk of manual errors
  • Security: Uses SSH to securely transfer the public key to the remote server

To use ssh-copy-id, run:

ssh-copy-id username@remote_host

This command will prompt for the remote user's password, then copy your public key to the server, enabling key-based authentication for future connections.

  • Note that the target host initially refused the connection. This was because I did not have an SSH server installed. Ensure that you have an SSH server installed.

    Here are examples of how to install an SSH server on different Linux distributions:

    • Ubuntu/Debian:
      sudo apt update
sudo apt install openssh-server
sudo systemctl enable ssh
sudo systemctl start ssh
    • CentOS/RHEL:
      sudo yum install openssh-server
sudo systemctl enable sshd
sudo systemctl start sshd
    • Fedora:
      sudo dnf install openssh-server
sudo systemctl enable sshd
sudo systemctl start sshd
    • Arch Linux:
      sudo pacman -S openssh
sudo systemctl enable sshd
sudo systemctl start sshd

    After installation, make sure to configure your firewall to allow SSH connections (typically on port 22) and review the SSH configuration file (/etc/ssh/sshd_config) for security settings.

  1. Select yes to continue the connection. You will then be prompted to enter the password of the remote machine.
  1. After successful authentication, you will see that the key was copied.
  1. Now we can test the connection with command ssh username@remote_host. I was prompted by Kali to enter my credentials to access the private key of the key pair.
  • SSH key pairs consist of a public key and a private key:
  • Public key: Stored on the remote server and used for encryption. It's safe to share.
  • Private key: Kept securely on the client machine and used for decryption. Never share this key.

The public key encrypts data that can only be decrypted by the corresponding private key. This system allows for secure, passwordless authentication between the client and server.

  1. Connection established successfully.

Troubleshooting SSH Key Pair Login Issues

If you encounter problems when trying to log in to a remote host using SSH key pair authentication, consider the following troubleshooting steps:

  • Check key permissions: Ensure that the private key file on the client has the correct permissions (typically 600 or -rw-------)
  • Verify public key on server: Confirm that the public key is correctly added to the ~/.ssh/authorized_keys file on the remote server
  • Check SSH server configuration: Review the /etc/ssh/sshd_config file on the server to ensure that PubkeyAuthentication is set to yes
  • Examine SSH logs: Look at the SSH server logs (often in /var/log/auth.log or /var/log/secure) for any error messages related to key authentication
  • Test with verbose output: Use the -v flag with ssh command (e.g., ssh -v user@host) to get detailed debugging information
  • Verify SSH agent: Ensure that the SSH agent is running and has the correct key loaded (use ssh-add -l to list loaded keys)
  • Check for network issues: Confirm that there are no firewall rules or network problems preventing the SSH connection

If problems persist after trying these steps, consider regenerating the key pair.
